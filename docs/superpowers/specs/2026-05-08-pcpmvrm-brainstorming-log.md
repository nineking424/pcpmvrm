## 0. 프로젝트 비전 (사용자)

> 리눅스 기반 파일처리 시스템 운영중인데, 대량의 파일을 옮기거나 복사할 일이 많아(수만~수백만). disk io는 남아도는데, 바닐라 cp와 mv, rm 명령은 병렬 수행 기능이 없어 불편해. 그렇다고 매번 gnu parallel이나 xargs를 사용해 스크립트를 짜기도 번거롭고 휴먼 에러 리스크도 있어. fpsync, rclone, fpart, rsync 등 많은 툴들이 있지만 cp, mv, rm을 모두 지원하지는 않는다거나 내부적으로 내 의도와 다르게 동작하는 등 부족한 부분들이 조금씩 있어. 이 프로젝트는 cp, mv, rm을 사용하는 유저들에게 최소 러닝커브와 최대 효율을 제공하는 것이 목표야.

**초기 방향성**:

- `pcp`, `pmv`, `prm` 세 개의 툴을 제작
- 각 툴은 바닐라 명령에 병렬 처리 기능을 더한 형태
- 공통 옵션 `--parallel`, 기본값 1
- 바닐라 시맨틱에 충실 — "p만 붙여서 실행"이 가능
- Go 언어 (goroutine 기반)

---

## Q1. 병렬화의 단위

**질문 요지**: `cp`/`mv`/`rm`은 다중 소스(`cp a b c d/`)와 재귀(`cp -r src/ dst/`) 모두 지원한다. 워커가 한 번에 가져가는 작업 단위를 어떻게 잡을 것인가?

| 옵션 | 설명 |
|---|---|
| A. 파일 단위 | 트리를 walk해서 모든 regular file을 큐잉, 워커가 1개씩 |
| B. top-level 인자 단위 | 인자 단위로 분배, 재귀는 워커 내부에서 직렬 |
| C. 하이브리드 | 자동 또는 옵션 |

**사용자 답변**:
- 여러 소스 인자(`pcp a b c dst/`) 지원하지 않음 (활용도 낮음)
- 재귀는 기본적으로 **A** 사용
- `--strict-order` 옵션을 주면 디렉토리 단위 병렬 + 디렉토리 내 메타 작업 순서 보장

---

## Q2. 워커가 작업을 수행하는 방식

**질문 요지**: native syscall 직접 구현 vs 자식 프로세스 호출. 호환성과 성능의 트레이드오프.

| 옵션 | 설명 |
|---|---|
| A. 자식 프로세스 호출 | 매 파일마다 `cp/mv/rm` fork+exec — 호환성 100%, fork 오버헤드 |
| B. Go syscall로 직접 구현 | 핵심 옵션만, 미지원은 거부 — 빠름, 엣지케이스 직접 챙김 |
| C. 하이브리드 | 핵심은 native, 미지원 옵션은 폴백 |

**사용자 답변**:
- **B**가 기본 방향 (핵심 옵션 직접 구현)
- 사용자가 `--force` 옵션을 주면 A 방식으로 작업 (이후 `--fallback`으로 명칭 변경 — Q9-1)
- `--force` 없이 미지원 기능 사용 시도하면 안내 후 종료. 안내 문구에 `--force`(→ `--fallback`) 사용 가능 안내와 성능 저하 경고 포함

**추가로 사용자가 제안한 신규 옵션**:
- `--strict-extension`: 특정 확장자를 가진 파일들만 메타 순서 보장
- 사용 시나리오: 트리거 파일(데이터)과 비정형 파일(이미지)이 섞여 있을 때, 비정형 대다수는 병렬로 효율 극대화하고 트리거 파일만 안전하게 처리

---

## Q3. `--strict-extension`의 정확한 의미

**질문 요지**: "메타 순서 보장"의 두 가지 해석을 좁힘.

| 해석 | 동작 |
|---|---|
| A. 그룹 내 순서 보장 | 대상 확장자끼리 lexical 직렬, 비대상과의 상대 순서는 미보장 |
| B. 트리거 시맨틱 (last-write barrier) | 비대상이 모두 끝난 후에야 대상 확장자가 시작, 대상은 lexical 직렬 |
| C. 양쪽 옵션 |  |

**부가 질문**: 다중 확장자(`.json,.csv`)의 그룹화.

**사용자 답변**:
1. **B (트리거 시맨틱)**
2. 다중 확장자 허용, **한 그룹**으로 묶어 phase 2에서 함께 직렬

## Q4. 직접 구현(B 모드) 옵션 범위

**질문 요지**: GNU `cp/mv/rm`의 옵션을 5개 티어로 분류, 어디까지 native로 구현할 것인가? 그리고 `-i`는 병렬과 본질적으로 충돌하므로 별도 결정.

| 티어 | 옵션 | 설명 |
|---|---|---|
| T1 필수 | `-r/-R`, `-f`, `-v` | 거의 모든 사용 사례 |
| T2 메타데이터 | `-p`, `-a`, `--preserve=...` | 백업·아카이브 |
| T3 충돌 처리 | `-n`, `-u` | 병렬에서 race 주의 |
| T4 심볼릭/하드링크 | `-L/-P/-H`, `-d`, `-l`, `-s` | 시맨틱 까다로움 |
| T5 고급 | `--reflink`, `--sparse`, `-x`, xattr/ACL | OS 의존성 큼 |

`-i` 처리 옵션: 거부 / 직렬화 다운그레이드 / 무시.

**사용자 답변**:
1. **B (T1+T2+T3 직접 구현)**, T4·T5는 `--fallback` 폴백
2. `-i`는 **거부** (`--fallback`으로만 사용 가능)

## Q5. 에러 처리 정책

**질문 요지**: 한 파일 실패 시 다른 워커들의 행동.

| 옵션 | 설명 |
|---|---|
| A. fail-fast | 첫 에러에 모든 워커 즉시 중단 — 바닐라 동작에 가까움 |
| B. best-effort | 에러는 로그하고 다른 파일 계속 처리, 종료 시 카운트와 exit code 보고 |
| C. 둘 다 + 플래그로 선택 |  |

**부가 질문**: 실패 로그 파일 정책 — 자동 / 명시 / stderr만.

**사용자 답변**:
1. **B를 기본**으로 진행, A는 **`--exit-on-error`** 옵션으로 활성화
2. **자동 로그를 기본**으로, **`--error-log=PATH`** 옵션으로 경로 지정 가능

## Q6. `pmv`의 cross-device 동작

**질문 요지**: `mv`는 같은 파일시스템 내에서는 `rename(2)` 한 번이지만, 다른 파일시스템 간에는 `EXDEV`로 cp+rm 폴백 필요. 병렬화 의미가 케이스별로 완전히 다름.

| 케이스 | 작업 단위 | 병렬화 가치 |
|---|---|---|
| same-device | `rename(2)` 1회 | 적음 (디렉토리 락 경합 가능) |
| cross-device | 데이터 복사 + 원본 삭제 | `pcp`와 동일, 큼 |

| 옵션 | 설명 |
|---|---|
| A. 자동 감지 + 자동 폴백 | 바닐라와 동일 |
| B. 명시적 거부 | 사용자가 직접 cp+rm 흐름 사용 |
| C. 자동 폴백 + `--no-cross-device` |  |

**부가 질문**: same-device일 때 워커 수 처리.

**사용자 답변**:
1. **A (자동 감지 + 자동 폴백)**
2. **자동 다운그레이드**, 워커 수 1로 강제

## Q7. 워크 큐 모델과 진행 상황 표시

**질문 요지**: 수백만 파일에서는 walk 자체가 수 분 걸림. 어떤 모델인가?

| 모델 | 설명 |
|---|---|
| A. Pre-walk | 시작 시 전체 walk → 큐에 다 넣고 처리 시작. 정확한 진행률·ETA, 메모리 부담 |
| B. Streaming | walk와 처리 동시 진행. 즉시 시작, 진행률은 처리량 기반 |
| C. Hybrid | streaming + 백그라운드 walk로 ETA 점진 정확화 |

**부가 질문**: 출력 인터리빙 — mutex 직렬화 / 버퍼링 후 일괄.

**사용자 답변**:
- **B (streaming) + mutex 직렬화**

진행 표시 형식:
```
[pcp]  243,182 files | 1.4 GB | 8,231 files/s | 47 MB/s | 4m12s elapsed | 12 errors
```

## Q8. `prm`의 안전장치

**질문 요지**: `rm`은 위험 명령이고 병렬은 그 위험을 가속한다. 어떤 안전장치가 필요한가?

| 옵션 | 설명 |
|---|---|
| A. 바닐라와 동일 | `--no-preserve-root`만 |
| B. 바닐라 + 대량 confirmation | N개 이상이면 시작 전 1회 확인 |
| C. 바닐라 + 보호 경로 목록 | `--protect=...` |
| D. B + C 조합 |  |

**사용자 답변**:
- **A (바닐라와 동일)**

---

## Q9. 시그널 처리 + Dry-run

**질문 요지**: 수십 분~수 시간 걸리는 작업 중 Ctrl+C 처리.

| 옵션 | 설명 |
|---|---|
| A. Hard stop | 즉시 종료, 부분 파일 남을 수 있음 |
| B. Graceful shutdown | 첫 SIGINT: 워커가 현재 Job 마치고 종료. 두 번째: 즉시 |
| C. Resumable | B + 처리 완료 목록 상태 파일에 기록 |

**부가 질문**: dry-run 지원 여부.

**사용자 답변**:
1. **B (graceful)**
2. dry-run **지원**

## Q9-1. `--force` vs `--fallback` 이름 충돌

**문제**: 바닐라 `cp -f`는 "overwrite", 우리의 `--force`는 "subprocess fallback". 이름이 겹쳐 사용자 혼란.

| 옵션 | 설명 |
|---|---|
| A. `--force` 그대로 | `man`에 명시 |
| B. `--fallback`로 변경 | 의미 명확 |
| C. `--subprocess` / `--compat` |  |

**사용자 답변**: **B (`--fallback`으로 변경)**

## 최종 결정 요약

| 항목 | 결정 |
|---|---|
| 명령 시그니처 | `pcp/pmv SRC DST`, `prm PATH` — 단일 소스 |
| 기본 워커 수 | `--parallel=1` |
| 워커 동작 모드 | Native syscall 직접 구현 |
| `--fallback` | 자식 프로세스 위임 모드 |
| 미지원 옵션 입력 | 안내 + exit (안내문에 `--fallback`과 성능 경고 포함) |
| 직접 구현 옵션 범위 | T1+T2+T3 |
| `-i` | 거부 |
| 병렬화 단위 (기본) | 파일 단위 |
| 병렬화 단위 (`--strict-order`) | 디렉토리 단위 + 내부 순서 보장 |
| `--strict-extension=LIST` | 트리거 시맨틱, 다중 확장자는 한 그룹 |
| 에러 정책 (기본) | best-effort |
| 에러 정책 (`--exit-on-error`) | fail-fast |
| 에러 로그 | 자동 + `--error-log=PATH` |
| `pmv` cross-device | 자동 감지 + 자동 폴백 |
| `pmv` same-device | 워커=1 강제 다운그레이드 |
| Walk 모델 | Streaming |
| 진행 표시 | 1초 단위, 처리량 기반 |
| 출력 인터리빙 | mutex 직렬화 |
| `prm` 안전장치 | 바닐라와 동일 |
| 시그널 처리 | 첫 SIGINT graceful, 두 번째 즉시 |
| `--dry-run` | 지원 |
| 디렉토리 mkdir/rmdir | Walker가 직접 처리 (워커 위임 안 함) |

상세 설계는 [`2026-05-08-pcpmvrm-design.md`](./2026-05-08-pcpmvrm-design.md) 참조.
